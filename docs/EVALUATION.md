# Оценка и evidence

## Режимы исполнения не смешиваются

- `live_ouroboros` — пакет сохранён новым реальным прогоном pinned runtime через оба
  MCP-инструмента; только это live-доказательство;
- `validation_only` — управляемые исходы B11–B13 без генерации;
- `deterministic_template` — явно помеченный не-LLM артефакт (разработка/резервный режим);
- `replay` — показ ранее захваченного immutable-артефакта;
- `mock` — только тесты.

`make eval-replay` — детерминированный оракул: он обязан проходить 15/15 при нуле provider-вызовов
и нуле live-кейсов и всегда сообщает `release_targets_passed=false`. X01–X05 измеряются отдельно
(`make test-chaos`) и не попадают в нормальные латентности и бизнес-метрики.

## Механическая и качественная оценка

**Механическая** — versioned-assertions на кейс (терминальное состояние, каналы, режим, точные
значения, QA), 22-check registry и hard-счётчики выше; ручной отзыв не переписывает машинный
результат.

**Качественная** — человек полностью читает шесть заранее выбранных live-пакетов (B01, B02, B03,
B04, B07, B08) и ставит оценки 1–5 по восьми пунктам: ясность; естественность русского языка;
соответствие сегменту/брифу; полезность персонализации; убедительность без давления;
нешаблонность; согласованность SMS и e-mail; готовность к показу. Формы в evidence намеренно
пустые (`WAITING_FOR_OPERATOR`) до реального ревью; test-actor и инженерные наблюдения не
засчитываются. Каждая запись ревью копирует опубликованный `packet_sha256` своего кейса.

## Guarded платный протокол

Перед полной корзиной: один warmup, один smoke и 2–3 разных пилота (`make gate2-live-pilot`,
`PILOT_CASE_ID` ∈ B04/B07/B08); фактические выходы читаются, затем `make live-readiness`
связывает их в checksummed readiness-манифест с консервативной проекцией (множитель ≥1.2).
Единственная поверхность полной корзины:

```bash
ALLOW_LIVE_EVAL=true \
EVALUATION_ID=<new-id> \
EVAL_PROVIDER_PROFILE=openrouter-glm-5.2-functional \
EVAL_MAX_TOKENS=<positive-approved-cap> \
EVAL_MAX_COST_USD=<positive-cap> \
EVAL_CONCURRENCY=1 \
make eval-live
```

Runner отклоняет повторные ID, непустые каталоги, грязный commit, несовпадающий strict-lock,
устаревший readiness и отсутствующие лимиты; выполняется строго последовательно, останавливается
между кейсами при нехватке headroom, сбои сохраняет и никогда не переключает модель/провайдера.
Без continuation-v4 отсутствие usage по-прежнему fail-closed; при активной v4-политике runner
классифицирует физический запрос, продолжает независимые кейсы и учитывает bounded estimate вне
provider ledger. `verify*`-команды не могут запустить платный прогон.

Для owner-authorized GLM resume текущий handoff дополнительно задаёт hash-bound session lineage.
Parent attempts входят в совокупный и фазовый расход, но их failed, полностью учтённый и
checksum-recovered model drift не является вечным запретом новой session. Исключение применяется
только к failed evidence и позволяет направленное исправление: новая attempt всё равно обязана
использовать exact OpenRouter `z-ai/glm-5.2`; completed drift, provider drift, missing usage или
несогласованный recovery остаются fail-closed. Лимиты попыток считаются для новой session, а
исторический spend не обнуляется.

OpenRouter physical-call accounting дополнительно связывает каждый полученный HTTP response header
с валидированным `X-Generation-Id`, terminal-событием клиента и одной usage-строкой. Исторические
pre-v4 whole-run reservations сохранены только как история и освобождены из новой authority.
Continuation-v4 начинает отдельное окно после hash-bound ledger prefix: generation-bearing orphan
опрашивается не более 600 секунд, затем резервируется только этот запрос как prompt upper bound +
configured max output с 2x ценовым запасом. Pre-generation failure без generation ID — anomaly с
нулевым reservation. Exact metadata заменяет estimate обычной provider-usage строкой; разрешённый
resolved alias GLM записывается отдельно, а ledger model остаётся `z-ai/glm-5.2`. Incomplete run
остаётся failed/non-evidence и повторяется новым linked ID; независимые basket cases продолжаются.

## Семантика frozen evidence

`EVALUATION_ID=<id> make evidence` принимает только существующий чистый checksum-bound frozen
live-источник на текущем clean commit. Собираются offline PDF/JPG/HTML/CSV/JSON/JSONL, по-кейсовые
записи, safe-трейсы, скриншоты, checksums и `IMMUTABLE.json`; replay-источник отклоняется. После
финализации каталог неизменен; человеческие записи (ревью/утверждения/sign-off/видео) образуют
производный пакет и не трогают исходные live-артефакты.

## Исторический отчёт Basket-03

Обезличенный [отчёт о тестировании MVP](../reports/basket03-mvp-testing/README.md) сохраняет
фактические входы и выходы Basket-03, QA, режимы, latency и usage. Результат — 13/15: B05 и B08
явно отмечены как неуспешные. Пакет связан с commit исполнения `0bdebf6` и рекомендуемой основой
кода `6010d93`, но остаётся `FAILED_NON_EVIDENCE` и не заменяет frozen release evidence.

## Что фактически есть и чего нет

- **Есть:** десять подтверждённых живых кейсов Basket-03 (B01, B02, B03, B04, B06, B07, B09,
  B10, B14, B15) с сохранёнными фактическими СМС/письмами, полным учётом usage и
  детерминированной оценкой 100/100 — они опубликованы на вкладке «Результаты» и в
  [отчёте Basket-03](../reports/basket03-mvp-testing/README.md).
- **Нет:** канонической frozen-корзины 15/15. Basket-03 в целом сохраняет технический статус
  `FAILED_NON_EVIDENCE` (13/15; B05 и B08 неуспешны из-за повторного вызова сохранения) и
  является честным отчётом тестирования MVP, а не release evidence.
- Любое изменение prompt/schema делает прежние readiness-записи непригодными: перед новой
  корзиной обязательны свежий review инструкции и новый readiness-набор.
- Независимые человеческие оценки, sign-off, репетиции и видео не выполнены; их статусы — в
  [чеклисте сдачи](SUBMISSION_CHECKLIST.md).

## Последовательность будущей переквалификации

Для текущего маршрута и для любого будущего изменения модели/провайдера порядок один и не
допускает срезания углов: сначала capability-probe (транспортный прогон обоих инструментов с
полным учётом usage), затем небольшая сложная мини-корзина из представительных кейсов
(B04/B07/B08 — точные значения, URL/UTM, UCS-2/emoji) с чтением фактических выходов, и только
после зелёного readiness — полная guarded-корзина B01–B15 с новым `EVALUATION_ID`. Совместимость,
качество, латентность и стоимость альтернативных маршрутов не заявляются до фактического
прохождения этой последовательности.
