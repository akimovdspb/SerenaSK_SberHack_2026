# Маршрут для жюри

Одна страница: проблема, решение, сквозной путь и соответствие шести критериям Project Results.
Каждый тезис опирается на проверяемый экран или файл репозитория.

## Проблема и пользователь

Маркетолог банка готовит персонализированные рассылки для бизнес-клиентов. Сегодня это ручной
процесс: собрать данные о продукте, сегменте и истории касаний, написать СМС и письмо, проверить
каждое число, ссылку, дисклеймер и ограничение канала, согласовать с утверждающим. Ошибка в факте
или запрещённое обещание стоят дороже самой рассылки. Пользователи прототипа — маркетолог (бриф,
замечания) и утверждающий (правила, пакет); все данные синтетические, внешняя отправка отключена.

## Решение за 60 секунд

Пользователь собирает свободный бриф и связывает его с отдельной факт-картой существующего или
нового синтетического продукта. Бриф задаёт аудиторию, задачу и тон; факт-карта — точное название,
CTA и единственные разрешённые продуктовые утверждения. Готовый контекст превращается в
проверенный комплект «СМС + письмо» одной управляемой задачей ГигаАгента (Ouroboros). Неполный
бриф детерминированно останавливается на уточняющих вопросах —
модель не вызывается. Агент получает только разрешённую версию фактов через первый MCP-инструмент,
сам пишет оба канала и один раз сохраняет типизированный черновик через второй. Backend без LLM
независимо сверяет каждое число, дату, ссылку и политику (22 автоматические проверки) и блокирует
утверждение при нарушении. Замечание человека создаёт новую видимую версию с точным diff;
повторяемое пожелание оформляется как декларативное правило с тестами и человеческим утверждением.
Отправки нет: результат — утверждённый человеком ZIP-экспорт.

## Сквозной путь на стенде

1. Вход по выданной команде ссылке → «Кампании и коммуникации» → «Новая кампания» или
   редактируемый пример «Пульс Выплат · финансовый менеджер».
2. Мастер разделяет существующий продукт и «Новый продукт». Второй путь требует строгой
   синтетической факт-карты и подтверждения отсутствия ПДн; objective/notes факты не пополняют.
3. Пользователь проверяет цель, аудиторию, событие, SMS/e-mail, тон и CTA. Пример переносит только
   бриф, никогда не переносит сохранённый output. После серверной валидации — «Создать через
   Ouroboros»:
   новая живая задача ГигаАгента, в трассе видны ровно два MCP-вызова (контекст → сохранение).
4. Комплект: СМС с метриками сегментов, письмо с безопасной вёрсткой, факт-карточка,
   привязка «утверждение → факт → источник», проверка качества.
5. В пустое поле вводится конкретное редакторское замечание → точечная версия 2: меняются только
   разрешённые пути, виден diff и полный повторный контроль качества.
6. Одно действие «Предложить правило» запускает отдельную управляемую задачу Ouroboros. После
   целевого, регрессионных и out-of-scope тестов человек утверждает ограниченную область правила.
7. Обычная новая кампания того же продукта получает активное правило из контекста; в её брифе нет
   выученной формулировки. Это демонстрирует перенос редакторского опыта без скрытой самоэволюции.
8. Утверждение пакета человеком (заблокировано при любом нарушении) → отдельный ZIP-экспорт.
9. Вкладка «Результаты» честно показывает десять исторических живых кейсов; новая prompt identity
   их не перегенерировала. B01–B15 находятся отдельно в «Тестовых сценариях» и не ограничивают
   рабочий authoring.

## Что подтверждено артефактами

**Подтверждено артефактами:** десять живых end-to-end кейсов на маршруте
`openrouter::z-ai/glm-5.2` внутри закреплённого ГигаАгента (Ouroboros `v6.61.4`) с сохранёнными
фактическими СМС/письмами, полным учётом токенов и детерминированной оценкой 100/100; полный
технический протокол Basket-03 (13/15, неуспешные B05/B08 сохранены, а не скрыты); детерминированные
проверки, сканы безопасности и браузерная матрица на контрольных точках разработки.
