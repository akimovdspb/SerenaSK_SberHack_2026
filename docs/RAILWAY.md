# Railway: рабочий hosted-профиль

Railway-профиль поднимает весь synthetic-only/no-send прототип одним сервисом и одним доменом.
Внутри контейнера отдельными процессами работают Caddy, приватный auth-сервис, FastAPI и
закреплённый Ouroboros `v6.61.4`; наружу открыт только Caddy. Это тот же продукт и та же кодовая
база, но дополнительный hosted-профиль, а не замена четырёхшаговому Docker Compose-запуску из
README.

Hosted-путь фактически проверен 18 июля 2026 года на runtime-checkpoint `2925715`: обычный мастер
создал новую кампанию, Ouroboros через `openrouter::z-ai/glm-5.2` сохранил live-комплект СМС и
письма, QA прошёл `22/22`, затем были проверены человеческое утверждение точного хэша и
ZIP-экспорт. В отдельном проверенном сценарии замечание редактора стало версией 2, ограниченным
правилом и применилось к следующей кампании. Эти демонстрационные прогоны не подменяют историческую
доказательную выборку из десяти кейсов Basket-03.

## Развёртывание

Репозиторий содержит npm workspace, поэтому автоматический импорт GitHub может ошибочно предложить
отдельный сервис `@communication-factory/web`. Он соберёт только frontend и не поднимет продукт.
Нужен один сервис из корня репозитория:

1. Создать **Empty Project**, затем **Empty Service** и подключить нужную GitHub-ветку.
2. В Settings оставить **Root Directory пустым**, Config File Path задать `/railway.toml`.
   Builder должен отображаться как `Dockerfile`, Dockerfile Path — `/Dockerfile`, Start Command —
   пустой.
3. В Variables не импортировать Suggested Variables из `.env.example`. Задать ровно три секрета:
   `APP_ACCESS_USERNAME`, `APP_ACCESS_PASSWORD`, `OPENROUTER_API_KEY`.
4. Сгенерировать публичный домен и дождаться в Deploy Logs строки
   `railway-bootstrap: READY provider=openrouter model=z-ai/glm-5.2`.

Значения логина и пароля выбирает владелец конкретного стенда. Они не хранятся в Git и не должны
публиковаться в README или презентации. Для проверки жюри владелец передаёт адрес и учётные данные
отдельным согласованным каналом. После запуска домен открывает отдельную страницу входа; локальный
Docker Compose создаёт случайные данные входа в `runtime/operator/access.txt`.

## Нужен ли Volume

Volume с mount path `/var/lib/communication-factory` **рекомендуется**, но не требуется для старта
и одного прогона. Он сохраняет между restart/redeploy:

- SQLite с кампаниями, версиями, правилами, решениями и ZIP-экспортами;
- рабочее состояние Ouroboros и результат executable review.

Без Volume сервис всё равно запускается и генерирует, но изменяемое состояние после redeploy может
исчезнуть, а платный review может потребоваться заново. Поэтому перед записью демо не следует
делать redeploy только ради очистки. Для чистого сценария используйте штатный сброс в
«Диагностике»: он не удаляет каталог продуктов, runtime, skill, provider key и доказательную
витрину.

## Runtime и ожидание

- Маршрут закреплён как `openrouter::z-ai/glm-5.2`; автоматической смены модели или fallback нет.
- Hosted-профиль даёт одной задаче до 600 секунд, внешнему run — до 900 секунд. Канонические
  Compose/evidence defaults от этого не меняются.
- В Railway включён максимум один контролируемый повтор только после типизированного временного
  transport/provider-сбоя. Повтор использует новый attempt ID, но тот же контекст, модель,
  prompt/skill/tool identity. Content/QA/schema failure повтор не разрешает.
- Если интерфейс сообщает «Временный сбой — повторяем запрос (2 из 2)», оператор ничего не
  подтверждает. Вторая неудача завершает именно этот запуск явной ошибкой; бесконечного цикла нет.
- Review инструкции является отдельным provider-вызовом. Сервис не перезапускает его бесконечно:
  после исправления причины один новый запуск разрешается изменением
  `AUTO_BOOTSTRAP_REVIEW_RETRY_ID`.

Provider-ключ launcher переносит из окружения Railway в закрытый файл процесса Ouroboros и
удаляет из окружения supervisor. FastAPI, auth-сервис и Caddy ключ не получают. Backend не имеет
другого LLM-клиента.

## Что проверять в продукте

### Обычная кампания

1. Войти через отдельную страницу и открыть «Кампании» → «Новая кампания».
2. Выбрать существующую синтетическую факт-карту или создать новую синтетическую карточку без ПДн.
3. Заполнить редактируемый бриф и нажать «Создать через Ouroboros».
4. Дождаться `live_ouroboros`, проверить самостоятельные СМС и письмо, факты, привязки, QA и
   безопасную трассу.
5. Оставить замечание к выбранному пути, получить неизменяемую версию 2 и diff.
6. Один раз запросить проект ограниченного правила, проверить область и тесты, затем утвердить его
   человеком.
7. Создать обычную новую кампанию того же продукта и проверить применение активного правила.
8. При необходимости утвердить точный хэш пакета и скачать ZIP. Внешней отправки всё равно нет.

### Доказательная витрина

Вкладка «Результаты» показывает ровно десять сохранённых живых кейсов Basket-03. Карточки
раскрывают фактические СМС/письма и не вызывают модель повторно. Отчёт проверяется по SHA-256 при
каждом чтении; при расхождении витрина закрывается с явной ошибкой. Полный каталог B01–B15 остаётся
отдельной regression-корзиной и не ограничивает обычный authoring.

## Безопасный сброс

В «Диагностике» нажать «Сбросить демо-состояние», ввести точную фразу `СБРОСИТЬ ДЕМО` и подтвердить
очистку. Сброс удаляет только пользовательские кампании, версии, решения, правила и ZIP текущего
стенда. Во время активного запуска он отклоняется. Frozen/исторические результаты, runtime,
provider key и каталог синтетических продуктов не затрагиваются.

## Быстрая диагностика

- `GET /healthz` → `200 ok` без сессии.
- Неавторизованный HTML-запрос → `303` на `/login`; API-запрос → `401`.
- После входа `GET /api/v1/health` возвращает `status: ok`, `environment: railway`,
  `data_mode: synthetic_only`, `external_send_enabled: false`.
- Cookie имеет `HttpOnly`, `Secure`, `SameSite=Lax`; изменяющие запросы проверяют CSRF.
- До `railway-bootstrap: READY` генерация может быть закрыта; причина видна в «Диагностике» и
  Deploy Logs.
- Во время записи ожидание ответа провайдера можно честно сократить монтажом, сохраняя один URL,
  видимые режим и идентификатор запуска и подпись «Ожидание живой генерации сокращено монтажом».

После любого изменения runtime/prompt/schema или hosted-конфигурации повторяются targeted-тесты,
desktop/narrow browser smoke и один свежий authoring-run. Исторические десять результатов при этом
не перегенерируются и не объявляются evidence новой identity.
